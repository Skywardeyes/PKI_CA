# 任务二执行手册

## 1. 适用范围

- 本手册适用于 **Windows**、**无 Docker**、**单人单机**（同一台电脑可轮流扮演管理员与用户）的本机演示。
- 目标：从 0 完成自建 CA、签发证书、双向 TLS 联调、成功/失败用例验证；可选使用 Web 控制台与 HTTP 证书仓库（CDP/AIA）辅助讲解。

## 2. 前置依赖

- OpenSSL（命令 `openssl` 可用）
- Python 3.10+（用于启动 Web 控制台）
- 浏览器（Edge/Chrome 任一）
- **无需**为 Web 控制台单独依赖 PowerShell；后端由 Python 直接调用 `openssl`。

建议先执行：

```powershell
openssl version
python --version
```

## 2.1 讲述口径（统一术语）

- **管理员**：执行建 CA、签发、吊销、审计查看的一方（单人演示时由你扮演）。
- **终端用户**：导入客户端证书并访问业务 HTTPS 的一方（单人演示时仍由你扮演）。
- **Web 控制台**：`web/` 提供的 FastAPI 管理页面（默认 `http://127.0.0.1:8765/`）。
- **Intl 区**：国际算法专区，API 前缀 `/api/intl/*`，产物目录 `ca/`、`server/`、`client/`。
- **GM 区**：国密算法专区，API 前缀 `/api/gm/*`，产物目录 `gm/ca/`、`gm/server/`、`gm/client/`。
- **HTTP 证书仓库**：`/repo/` 公开只读地址，用于分发 CA 链与 CRL（CDP/AIA 目标）。
- **业务 mTLS 服务**：`openssl s_server` 启动的本机服务（默认 `https://localhost:8443/`），用于演示双向 TLS。
- **审计日志**：`artifacts/logs/audit.jsonl`，记录管理员侧关键操作留痕。

## 8. CDP / AIA 与自检

- Intermediate 模板 [ca/openssl-intermediate.cnf](../ca/openssl-intermediate.cnf) 为 **服务端证书、客户端证书** 写入 **CDP** 与 **AIA（caIssuers）**，默认指向：`http://127.0.0.1:8765/repo/intermediate.crl.pem` 与 `http://127.0.0.1:8765/repo/ca-chain.cert.pem`。
- **改端口或主机名**后：已签发证书内 URI **不会变**，须改 `openssl-intermediate.cnf` 后 **重新签叶子证书**（或回到第 4 节整包清零重来）。
- **自检**（完成 7.3 后）：

```powershell
openssl x509 -in server/server.cert.pem -noout -text
```

在输出中确认存在 **CRL Distribution Points** 与 **Authority Information Access**。

- **仓库页**：浏览器打开 `http://127.0.0.1:8765/repo/`（须先按第 6 节启动 Web）。吊销后刷新 CRL 链接可看到更新。

## 9. 命令行 mTLS 联调（推荐先做）

### 9.1 终端 A：启动 mTLS 服务端

```powershell
openssl s_server -accept 8443 -www -cert server/server.cert.pem -key server/server.key.pem -CAfile ca/intermediate/certs/ca-chain.cert.pem -Verify 1 -verify_return_error
```

- 该终端会持续占用，属正常现象；可加 `-naccept 1` 只处理一次连接。

### 9.2 终端 B：成功用例（带客户端证书）

```powershell
cmd /c "echo Q | openssl s_client -connect 127.0.0.1:8443 -cert client/client-trainee.cert.pem -key client/client-trainee.key.pem -cert_chain ca/intermediate/certs/intermediate.cert.pem -CAfile ca/intermediate/certs/ca-chain.cert.pem -brief -verify_return_error"
```

预期：`CONNECTION ESTABLISHED`、`Verification: OK`。

### 9.3 终端 B：失败用例（不带客户端证书）

```powershell
cmd /c "echo Q | openssl s_client -connect 127.0.0.1:8443 -CAfile ca/intermediate/certs/ca-chain.cert.pem -brief -verify_return_error"
```

预期：客户端 `alert certificate required`；服务端 `peer did not return a certificate`。

## 11. 吊销与 CRL 失败用例

| 项目 | 内容 |
|------|------|
| **执行** | Web 点击 `5. 吊销客户端证书`（`clientName=trainee`） |

```powershell
openssl s_server -accept 8443 -www -cert server/server.cert.pem -key server/server.key.pem -CAfile ca/intermediate/certs/ca-chain.cert.pem -Verify 1 -verify_return_error -CRL ca/intermediate/crl/intermediate.crl.pem -crl_check
```

再用已吊销证书访问 `https://localhost:8443/`：预期 `certificate revoked` / 握手失败。

## 12. Web 控制台与单人演示顺序（摘要）

完整安装、令牌策略、公开 `/repo/` 说明见 [web/README.md](../web/README.md)。

Web 后端已做结构化改造：
- Intl 编排在 `web/app/services/pki_workflow.py`，GM 编排在 `web/app/services/pki_workflow_gm.py`
- 路由按 profile 拆分：`/api/intl/*` 与 `/api/gm/*`（旧 `/api/*` 兼容 Intl）
- JSON 接口统一返回 `ok / code / message / data / logs`；`data` 中含 `profile`、`steps`、`duration_ms`、`artifacts`（下载接口除外）

当前页面已接入 Intl/GM 双专区按钮（与各自 workflow 函数对应）：
- 演示重置 → `reset_demo()`
- 初始化目录 → `init_structure()`
- 建 CA → `build_ca()`
- 签发 → `issue()`
- 吊销 → `revoke()`
- 链路校验 → `verify()`
- 一键 mTLS 验证（TLS1.2）→ `mtls_validate(tls_version="tls1_2")`（**内含吊销**）
- TLS 握手观测（TLS1.2）→ `tls_observe(tls_version="tls1_2")`
- 一键 mTLS 验证（TLS1.3）→ `mtls_validate(tls_version="tls1_3")`
- TLS 握手观测（TLS1.3）→ `tls_observe(tls_version="tls1_3")`
- GM TLS1.3 为严格判定：任一握手失败即 `ok=false` / `OPENSSL_FAILED`

**建议现场顺序（管理员 → 仓库 → 用户）**：

1. 启动 Web：`uvicorn` 监听 `127.0.0.1:8765`（与证书内 CDP 端口一致）。
2. 以 **管理员** 在控制台完成：演示重置 → 初始化 → 建 CA → 签发 → 验证（可选填「操作备注」）。
3. **新标签** 打开 `http://127.0.0.1:8765/repo/`，结合第 8 节讲 CDP/AIA。
4. 切换 **用户**：导入 P12；另开终端 `openssl s_server`；浏览器完成第 10 节与失败用例。
5. 需要时「查看审计」：`artifacts/logs/audit.jsonl` 或 API `GET /api/audit/tail`（若设置了 `PKI_WEB_TOKEN` 需带头，见 web/README）。

## 12.1 TLS 三阶段与本项目映射（严格版）

- **第一阶段：TCP 三次握手**
  - 由内核网络栈完成，不直接写入项目日志。
  - 在本项目中，`openssl s_client` 能发起连接即代表 TCP 通道已建立。
- **第二阶段：TLS 握手**
  - 服务端证书验证：见 `artifacts/logs/tls-observe-tls12-with-cert.log`、`artifacts/logs/tls-observe-tls13-with-cert.log`
  - 客户端证书认证：见 `artifacts/logs/tls-observe-tls12-no-cert.log`（无证书失败）
  - 密钥交换与会话建立：建议对照 TLS1.2 与 TLS1.3 两份日志讲解差异
- **第三阶段：加密应用数据传输**
  - 握手成功后进入应用数据阶段；在本项目中可继续通过浏览器访问 `https://localhost:8443/` 观察业务响应。

**严格口径说明**：
- 培训“教材模型”常以 TLS1.2 + RSA 传输 Pre-Master Secret 讲解；
- 现代生产默认更常见 TLS1.2/1.3 + (EC)DHE（前向保密），本项目通过 `tls_observe()` 同时提供两类观测证据；
- 吊销后拒绝连接见 `artifacts/logs/tls-observe-tls12-revoked.log`。

## 13. 常见问题排查

- **端口占用**：`8443` 被占用可改为 `9443`，`s_client` 同步改端口。
- **证书与私钥不匹配**：出现 `key values mismatch` 时，重新签发服务端证书并保证 key/cert 成对。
- **文件不存在**：确认已按第 7 节顺序执行。
- **浏览器无证书**：确认导入到「当前用户 → 个人」且为 `.p12`。
- **时间错误**：系统时间异常会导致证书未生效/已过期。
- **CDP 打不开**：确认 Web 已启动且为 `8765`；访问 `http://127.0.0.1:8765/repo/`。
